# Codex 专业提示词：ROS2 RoboMaster S1 AI Control Deck 全新开发版 v6

> 适用场景：本提示词用于交给 Codex 重新规划并从零开发一个基于 ROS2 的 RoboMaster S1 Web 遥控、视频预览、目标识别、人物锁定和后续自动跟随系统。  
> 重要变化：本版本明确要求**不复用旧代码**、**机器人信息自动读取**、**页面全屏自适应**、**界面中文化**、**运行前提写入手册**、**远程测试服务器部署**、**每阶段代码推送 GitHub**、**每阶段必须输出部署与测试说明**、**明确三种摇杆控制模式语义**。

---

## 0. 最重要的开发原则

请严格遵守以下原则：

1. **这是一次全新开发，不是修补旧项目。**
   - 之前的 `ROS2 + yolo_ros + person_following_robot + 服务器后台推理` 代码不要继续复用。
   - 旧代码只允许作为需求背景、历史问题和硬件连接信息参考。
   - 不允许复制旧前端页面、旧后端结构、旧视频链路、旧跟随控制逻辑。
   - 如果当前目录存在旧代码，应先进行目录审计，明确哪些文件废弃、哪些配置可参考，并写入 `docs/LEGACY_REVIEW.md`。
   - 新代码应使用清洁目录结构重新组织。

2. **开发之前必须先读取项目目录中的 `AGENTS.md`。**
   - 在执行任何代码修改前，先读取并遵守 `AGENTS.md` 中的开发规范、目录约束、编码风格、测试要求和提交要求。
   - 如果 `AGENTS.md` 中的要求与本提示词冲突，以更安全、更可验证、更利于分阶段交付的要求为准。
   - 如果发现冲突，必须记录到 `docs/AGENTS_CONFLICTS.md`，说明冲突点和采用方案。

3. **不要再做“大而全的一条链路”。**
   - 禁止把视频回传、YOLO 推理、人物跟随、车辆控制全部耦合到一个后台闭环里。
   - 必须拆成四条独立链路：
     - 视频链路
     - 控制链路
     - AI 识别链路
     - 跟随控制链路
   - 每条链路都必须能单独启动、单独测试、单独排障。

4. **优先级必须固定。**
   ```text
   视频清晰流畅 > 手动遥控稳定 > ROS2 自动发现 > 目标识别 > 人物锁定 > 半自动辅助跟随 > 全自动跟随
   ```

5. **机器人信息必须由系统自动读取。**
   - 前端不应该要求用户手动填写摄像头地址、ROS2 Topic、机器人控制 Topic、云台 Topic 等基础信息。
   - 系统应通过后端 ROS2 自动发现能力读取机器人环境信息，并自动生成前端可用的连接配置。
   - 如果自动识别失败，也应展示“未发现原因”和“排查步骤”，而不是让用户直接填写一堆技术参数。

6. **页面必须在浏览器 100% 缩放下完整可见。**
   - 当前页面需要浏览器缩放到 75% 才能看全，这是不可接受的。
   - 新页面必须在常规浏览器 100% 缩放比例下适配：
     - 1920×1080
     - 1600×900
     - 1366×768
     - 笔记本浏览器窗口
   - 不得依赖浏览器缩放解决布局问题。

7. **界面语言必须是中文。**
   - 页面标题、按钮、状态、提示、错误信息、阶段说明、空状态说明必须使用中文。
   - 专业名词可以保留英文或中英混排，例如 ROS2、Topic、Node、Service、WebSocket、WebRTC、MJPEG、H.264、FPS、YOLO、cmd_vel、QoS。
   - 非专业名词禁止使用英文，例如不要显示 `Stream Offline`、`Connect`、`Disconnected`、`Phase Notes`，应改为“视频离线”“连接”“未连接”“阶段说明”。

8. **测试服务器是远程服务器，不在本地。**
   - Codex 不要假设用户会在 Codex 本机或本地浏览器完成最终联调。
   - 每个阶段开发完成后，必须把代码提交并推送到 GitHub 项目：`https://github.com/lgp2013/robomaster-s1-ai-system.git`。
   - 用户会在远程 Ubuntu 20.04 测试服务器上自行 `git pull` 后部署和测试。
   - 因此，每个阶段上传完成后，必须同步把远程服务器部署方法、运行命令、测试步骤、正常结果和排查方法写入手册。
   - 不允许只说“本地运行成功”，必须给出远程服务器可执行的部署和运行步骤。

9. **GitHub 是阶段交付的唯一代码同步入口。**
   - 每个阶段完成后必须执行 `git status`、`git add`、`git commit`、`git push`。
   - 不允许要求用户手动复制代码文件到远程服务器。
   - 不允许强制推送，不允许覆盖远程历史，不允许提交密钥、Token、`.env`、日志、大模型权重、临时缓存。
   - 如果没有 GitHub 写入权限，必须停止并明确说明原因，同时把需要用户执行的认证方式和命令写入手册。

## 1. 当前问题判断

当前方向的问题不是单纯代码 bug，而是原方案把多个实时任务放在同一条链路里，导致性能和稳定性不可控：

1. 视频回传要求低延迟、高帧率、清晰度高。
2. Web 遥控要求控制指令稳定、延迟低、可急停。
3. YOLO 推理会占用 CPU/GPU，不能阻塞视频和控制。
4. 自动跟随需要目标跟踪、云台控制、底盘控制、安全策略联合工作，不能在第一阶段实现。
5. 前端页面现在存在两个明显问题：
   - 需要用户手动填写 Stream URL，不符合 ROS2 自动发现能力。
   - 页面比例过大，需要浏览器缩放到 75% 才能看全，不符合生产级遥控界面要求。

因此，本项目必须从“视频和控制底座”开始，逐步叠加 AI 能力。

每个阶段完成后，必须执行并记录：

```bash
git status
```

## 6. 推荐总体技术栈

### 5.1 前端

推荐：

```text
React + TypeScript + Vite
Canvas / SVG 覆盖层
WebSocket 客户端
响应式 CSS
```

要求：

1. 页面必须是生产级遥控驾驶舱，不是后台管理页面。
2. 不要做普通表格式管理界面。
3. 不要使用 generic AI 风格。
4. 不要使用常见紫色渐变白底模板。
5. 不要使用 Arial、Roboto、Inter 等过于普通的默认字体。
6. 必须选择一个明确、强烈、统一的视觉方向。
7. 推荐视觉方向：
   ```text
   深色工业控制台 + RoboMaster 机甲驾驶舱 + 高对比黄色安全色 + 视频优先布局
   ```
8. 所有界面文案必须中文化。
9. 不要让用户填写 Stream URL。
10. 页面必须自动读取后端返回的机器人信息、视频状态、Topic 映射和控制能力。

### 5.2 后端

推荐：

```text
Python 3.8
FastAPI
WebSocket
rclpy
ROS2 Foxy
```

后端职责：

1. 自动发现 ROS2 环境。
2. 自动读取 ROS2 Node、Topic、Service、Parameter。
3. 自动识别摄像头 Topic。
4. 自动识别底盘控制 Topic。
5. 自动识别云台控制 Topic 或服务。
6. 提供前端 API。
7. 提供 WebSocket 状态推送。
8. 提供控制命令转发。
9. 提供安全控制策略。
10. 不承载高负载 YOLO 推理主链路，AI 节点独立运行。

### 5.3 视频链路

第一阶段：

```text
ROS2 Image Topic / CompressedImage Topic
→ 后端自动发现
→ 后端生成可访问的视频预览地址
→ 前端自动连接
```

可接受的第一阶段方案：

```text
MJPEG 预览
CompressedImage 预览
本地模拟视频源
```

后续生产方向：

```text
WebRTC
H.264
RTSP to WebRTC
GStreamer
硬件编码
```

强制要求：

1. 第一阶段可以用 MJPEG 建立可用基线。
2. 不允许把最终方案锁死在 MJPEG。
3. 不允许把 1080P 原始图像通过 rosbridge JSON WebSocket 推给前端。
4. 视频链路必须独立于 YOLO 推理链路。
5. 视频卡顿时，不能影响手动控制。
6. YOLO 推理卡顿时，不能影响视频预览。

### 5.4 控制链路

推荐：

```text
前端虚拟游戏手柄
→ WebSocket
→ 后端 ROS2 控制桥接
→ cmd_vel / 云台控制 Topic / RoboMaster SDK 封装节点
```

#### 5.4.1 摇杆硬件映射必须固定

本项目的遥控交互不是普通后台按钮，也不是十字方向键。必须按照“掌机 / 游戏手柄”的直觉进行设计和实现：

1. **左摇杆 = 底盘摇杆。**
   - 对应车辆底盘移动。
   - 前后方向控制线速度 `linear.x`。
   - 左右方向控制角速度 `angular.z` 或底盘转向。
   - UI 标签必须明确显示为“底盘摇杆 / 底盘控制”。

2. **右摇杆 = 云台摇杆。**
   - 对应云台转动。
   - 左右方向控制云台偏航 yaw。
   - 上下方向控制云台俯仰 pitch。
   - UI 标签必须明确显示为“云台摇杆 / 云台控制”。

3. 前端视觉布局必须体现这个映射：
   - 左侧操作区域、左物理摇杆、左虚拟摇杆都表示底盘。
   - 右侧操作区域、右物理摇杆、右虚拟摇杆都表示云台。
   - 不得把两个摇杆都放在右侧后再要求用户理解映射。
   - 不得用“Joystick 1 / Joystick 2”等英文或模糊命名。

#### 5.4.2 三种控制模式的真实语义

必须严格按照下面的控制模式实现，不允许再使用模糊描述。

| 控制模式 | 可操作对象 | 底盘是否移动 | 云台是否移动 | 正确行为 |
|---|---|---:|---:|---|
| 底盘模式 | 左摇杆 / 底盘摇杆 | 是 | 否 | 只控制车辆底盘移动，云台保持当前姿态不动。右摇杆应置灰或标记“云台未启用”。 |
| 云台模式 | 右摇杆 / 云台摇杆 | 否 | 是 | 只控制云台偏航和俯仰，车辆底盘不移动。进入该模式时必须向底盘发送一次零速度，防止误动作。 |
| 联动模式 | 右摇杆 / 云台摇杆为主控 | 是 | 是 | 用户操作云台摇杆时，云台先按摇杆方向转动，车辆底盘跟随云台当前指向方向移动或转向。左摇杆默认禁用，避免与联动控制冲突。 |

#### 5.4.3 联动模式实现要求

联动模式的含义是“云台指向哪里，车辆跟随哪里”，不是简单同时发送两个无关命令。Codex 必须实现为独立、可测试的控制策略：

1. 右摇杆仍然是云台摇杆：
   - 横向输入控制云台偏航 yaw。
   - 纵向输入控制云台俯仰 pitch。

2. 后端必须维护云台当前指向状态：
   - 优先读取 ROS2 云台姿态反馈 Topic / Service / SDK 状态。
   - 如果暂时没有姿态反馈，可使用“最近一次云台控制目标值”作为临时估算，并在文档中说明这是临时实现。

3. 底盘跟随策略：
   - 根据云台偏航方向计算底盘目标航向。
   - 底盘通过 `cmd_vel` 或 RoboMaster SDK 控制逐步转向云台指向方向。
   - 允许低速前进，但必须限速；不得突然高速冲出。
   - 如果无法安全确定前进速度，第一版联动模式可以先实现“底盘自动转向对齐云台方向”，并在 UI 中明确显示“联动前进未启用”。

4. 安全要求：
   - 右摇杆松开后，云台命令归零，底盘命令也必须归零或进入安全保持。
   - 页面失焦、WebSocket 断开、心跳超时、急停触发时，底盘和云台都必须停止。
   - 联动模式必须有明显中文状态提示，例如“联动模式：车辆跟随云台方向”。
   - 如果进入联动模式后没有发现云台姿态反馈，必须提示“云台姿态未发现，使用临时估算”或“联动不可用”，不能静默误控。

#### 5.4.4 控制链路通用要求

1. 不使用十字方向键作为主控制方式。
2. 使用游戏手柄式虚拟操作杆。
3. 底盘移动和云台控制在代码结构上必须分离，不能把所有逻辑写在一个前端函数里。
4. 支持三种操作模式：
   - 底盘模式
   - 云台模式
   - 联动模式
5. 必须实现：
   - 控制心跳
   - 超时急停
   - 速度限制
   - 加速度限制
   - 松手自动归零
   - 页面失焦停止
   - WebSocket 断开停止
   - 急停按钮
6. 可参考 `teleop_twist_joy` 的思想，将 joystick 输入转换为 `Twist` 控制命令。
7. 必须把三种模式的控制映射、Topic 映射、正常结果写入 `docs/CONTROL.md`。

### 5.5 AI 识别链路

AI 识别不在第一阶段实现。

后续阶段要求：

1. YOLO 推理独立为 ROS2 节点。
2. 不阻塞视频链路。
3. 不阻塞控制链路。
4. 初期只识别两类：
   - 人
   - 障碍物 / 物体
5. 识别结果只传轻量结构化数据：
   ```json
   {
     "class": "person",
     "confidence": 0.92,
     "bbox": [x, y, w, h],
     "track_id": 1,
     "timestamp": 123456789
   }
   ```
6. 前端只叠加识别框，不接收大图推理结果。

### 5.6 自动跟随链路

自动跟随不在第一阶段实现。

后续应拆成独立节点：

```text
target_lock_node
target_tracker_node
gimbal_tracker_node
follow_controller_node
safety_arbiter_node
alert_node
```

跟随要求：

1. 只能锁定人，不能锁定墙、床、柜子等静态物体。
2. 锁定人物时需要拍照保存。
3. 跟随距离目标约 1 米。
4. 人物前进，车辆前进。
5. 人物后退，车辆后退。
6. 丢失人物后，先控制云台上下左右搜索。
7. 根据锁定照片进行目标重识别。
8. 10 秒未找回目标，则提示音告警、灯光闪烁。
9. 必须有最小安全距离。
10. 必须有障碍物停车策略。
11. 不允许在没有安全策略的情况下让机器人自动移动。

## 7. ROS2 自动发现机制

这是本版本最重要的修改点之一。

前端不应让用户填写机器人基础信息。后端必须提供 ROS2 自动发现能力。

### 6.1 自动发现启动流程

后端启动时执行：

```bash
source /opt/ros/foxy/setup.bash
# 如果存在项目工作空间
source install/setup.bash || true
```

然后执行或通过 rclpy API 完成以下发现：

```bash
ros2 node list
ros2 topic list -t
ros2 service list -t
ros2 param list
ros2 action list -t || true
```

并生成：

```text
runtime/robot_discovery.json
```

### 6.2 自动发现内容

必须识别以下信息：

1. ROS2 发行版。
2. 当前可见 Node。
3. 当前可见 Topic。
4. 当前可见 Service。
5. 摄像头 Topic 候选：
   - `sensor_msgs/msg/Image`
   - `sensor_msgs/msg/CompressedImage`
6. 控制 Topic 候选：
   - `geometry_msgs/msg/Twist`
   - 常见名称：`/cmd_vel`、`/cmd_vel_raw`、`/robot/cmd_vel`
7. 云台控制候选：
   - 与 gimbal、camera、pitch、yaw 相关的 Topic 或 Service。
8. 状态 Topic 候选：
   - 电池
   - 里程计
   - IMU
   - 机器人状态
   - 云台状态
9. 视频 Topic 的分辨率、帧率、最近帧时间。
10. 控制 Topic 的消息类型和是否可发布。

### 6.3 后端 API

至少提供以下接口：

```text
GET /api/health
GET /api/ros/status
POST /api/ros/rescan
GET /api/robot/discovery
GET /api/robot/capabilities
GET /api/video/sources
GET /api/video/status
POST /api/control/mode
POST /api/control/estop
WebSocket /ws/status
WebSocket /ws/control
```

### 6.4 前端行为

前端启动后必须：

1. 自动调用 `/api/robot/discovery`。
2. 自动显示机器人连接状态。
3. 自动显示 ROS2 环境状态。
4. 自动显示发现到的视频源。
5. 自动连接默认视频源。
6. 如果发现多个视频源，显示中文选择项，例如：
   ```text
   已发现 3 个视频源，请选择用于预览的视频源。
   ```
7. 如果没有发现视频源，显示：
   ```text
   未发现可用摄像头 Topic。请检查 ROS2 环境是否已 source，摄像头节点是否已启动。
   ```
8. 提供“重新扫描机器人”按钮。
9. 不显示让普通用户输入 Stream URL 的输入框。
10. 如确实需要调试手动 URL，应放在“高级调试”折叠面板中，并默认隐藏。

## 8. 前端页面设计要求

### 7.1 视觉方向

页面必须符合以下方向：

```text
RoboMaster 机甲驾驶舱
FPV 遥控画面
深色工业控制台
高对比安全黄色
视频画面是绝对主角
```

避免：

1. 普通后台管理页面。
2. 左侧菜单 + 表格 + 卡片堆叠。
3. 紫色渐变白底 AI 模板。
4. 大面积无意义 hero 标题。
5. 让标题挤占视频区域。
6. 需要滚动或缩放才能看到核心按钮。

### 7.2 页面布局强约束

必须解决当前页面比例问题。

要求：

1. 页面根容器必须适配视口：
   ```css
   width: 100vw;
   min-height: 100dvh;
   max-width: 100vw;
   overflow: hidden;
   ```
2. 不允许写死超大宽度或高度。
3. 不允许使用导致 1366×768 无法显示核心区域的巨大标题。
4. 页面在浏览器 100% 缩放下必须完整显示：
   - 视频区域
   - 连接状态
   - 主要操作按钮
   - 右侧状态摘要
5. 主要操作按钮不得被屏幕底部遮挡。
6. 视频区域优先级最高，占据主要空间。
7. 右侧状态栏在小屏下可折叠或移动到底部。
8. 字体大小使用 `clamp()` 自适应。
9. 视频容器使用 `aspect-ratio: 16 / 9`，但不得撑破视口。
10. 页面布局建议：
    ```text
    顶部：紧凑状态栏
    中间：左侧大视频 + 右侧状态/控制
    底部：模式切换、急停、手柄控制
    ```
11. 禁止因为页面美观牺牲可用性。
12. 必须在以下尺寸测试截图或记录：
    - 1920×1080
    - 1600×900
    - 1366×768
    - 浏览器窗口非全屏

### 7.3 中文界面要求

所有页面文案必须中文化。

### 7.4 第一阶段页面不应出现的内容

第一阶段是视频预览基线，因此页面可以保留后续功能入口，但必须明确禁用。

3. 摄像头 Topic 选择。
4. 重新扫描机器人。

| 联动模式安全释放 | 联动模式下松开右摇杆 | 云台命令归零，底盘停止或进入安全保持 |

| 页面失焦 | 任意模式操作时切换浏览器窗口 | 底盘和云台自动停止 |
| WebSocket 断开 | 模拟断开后端连接 | 底盘和云台自动停止 |

正常结果：

```text
左摇杆明确控制底盘
右摇杆明确控制云台
底盘模式下只发布底盘移动命令，不发布云台运动命令
云台模式下只发布云台运动命令，底盘保持零速度
联动模式下操作右摇杆，云台改变方向，底盘跟随云台方向移动或转向
松开摇杆后，底盘和云台命令归零
点击急停后，机器人立即停止
切换模式后，上一模式不会残留运动命令
```
